# CLIENT_DELIVERY_OS.md

## 1. Status
Active - Specialist OS

---

## 2. Purpose
Client Delivery OS defines how work is delivered after a deal is agreed.

It ensures:

- clear onboarding
- structured delivery
- consistent outcomes
- strong client experience

It sits after Sales OS.

---

## 3. Core Principle

Clear delivery creates trust.

Delivery must be:

- structured
- predictable
- easy to manage

---

## 4. Activation Conditions

Claude should suggest Client Delivery OS when:

- a prospect becomes a client
- work has been agreed
- delivery needs to begin
- the user needs help structuring client work

Claude must ask before activating.

---

## 5. System Flow

Deal Sourcing OS
-> Sales OS
-> Client agrees
-> Client Delivery OS
-> structured delivery
-> outcome
-> feedback loop

---

## 6. Core Functions

Client Delivery OS supports:

- client onboarding
- delivery planning
- task tracking
- communication structure
- expectation management
- progress tracking
- completion and handoff

---

## 7. Onboarding Layer

Ensures:

- client understands what happens next
- scope is clear
- expectations are set
- first steps are defined

---

## 8. Delivery Structure

Defines:

- what work is done
- how it is done
- in what order
- how progress is tracked

---

## 9. Communication Layer

Ensures:

- regular updates
- clear messaging
- no confusion
- alignment with client expectations

---

## 10. Progress Tracking

Tracks:

- tasks completed
- work in progress
- blockers
- delivery timeline

---

## 11. Boundaries

Client Delivery OS must NOT:

- change agreed scope without clarity
- promise outcomes not defined
- replace Sales OS decisions
- introduce new work without agreement

---

## 12. DCoS Roles

ChatGPT DCoS:
- builds delivery workflows
- creates onboarding steps
- structures task plans

Claude DCoS:
- reviews clarity
- checks risks
- ensures logic consistency

Claude controls output.

---

## 13. Output Rules

All outputs must be:

- structured
- clear
- easy to follow
- actionable
- aligned with user expectations

---

## 14. Memory Contribution

May store:

- successful delivery patterns
- common issues
- client feedback insights

Only useful patterns are stored.

---

## 15. Success Criteria

Client Delivery OS is working when:

- onboarding is smooth
- delivery is structured
- clients understand progress
- work is completed clearly
- expectations are met

---

## 16. Failure Conditions

Fails if:

- delivery is unclear
- communication breaks down
- tasks are untracked
- expectations are misaligned
- scope drifts

---

## 17. End State

Client Delivery OS is working when:

agreement
-> structured delivery
-> clear outcome
-> satisfied client

with no confusion or chaos.
