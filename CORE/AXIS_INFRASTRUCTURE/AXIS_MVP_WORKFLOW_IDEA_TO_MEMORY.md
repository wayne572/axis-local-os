# Axis MVP Workflow: Idea To Memory

Status: active - first complete v4 operating loop
Purpose: Define one complete end-to-end workflow that proves Axis as governed operational infrastructure.

## Workflow Summary

```text
Idea
-> structured plan
-> execution workflow
-> review and reflection
-> tracker update
-> memory update
```

One working loop is more valuable than many disconnected modules.

## Trigger

Use this workflow when Wayne says:

- "I have an idea"
- "Capture this"
- "Turn this into a plan"
- "Build this into a workflow"
- "Make this operational"
- "What should we do with this?"

## Step 1 - Capture Idea

Record:

- idea summary
- source or context
- related venture or product
- intended outcome
- urgency
- known constraints

Do not overbuild at capture stage.

## Step 2 - Classify

Classify by mode:

| Mode | Use |
|---|---|
| ORCHESTRATE | priority, decision, routing |
| BUILD | asset, workflow, product, system |
| LEAD | outreach, sales, pipeline |
| LEARN | research, intelligence, market scan |
| REVIEW | audit, risk, validation, signoff |

Also classify by risk:

- low: internal note or plan
- medium: affects workflow, offer, or client delivery
- high: affects pricing, compliance, external commitments, or public claims

## Step 3 - Structured Plan

Create a plan with:

- objective
- owner
- lifecycle state
- output required
- required source files
- specialist support needed
- review checkpoint
- next action

If unclear, ask one focused question.

## Step 4 - Execution Workflow

Build or execute only the next useful part.

For internal work:

- produce the usable artefact
- mark review need if relevant

For client-facing or external work:

- draft first
- run validation
- keep in review until approved

For system changes:

- document why
- update the correct core file
- avoid uncontrolled sprawl

## Step 5 - Review

Run the relevant review:

- Output Validation for quality and readiness
- Governance check for risk and authority
- Pricing Authority check for commercial claims
- Client AI Governance Pack for data and review boundaries
- Precision Alpha Risk Officer for financial intelligence

Review result must be:

- pass
- revise
- blocked
- escalate to Wayne

## Step 6 - Reflection

Capture:

- what worked
- what was unclear
- what repeated pattern appeared
- what should change next time
- whether memory should be updated

Reflection should be short and useful.

## Step 7 - Tracker Update

Update or propose update when:

- lifecycle state changed
- owner changed
- next action changed
- blocker appeared
- proof was created
- client/project status changed

Use existing trackers first.

## Step 8 - Memory Update

Memory update is allowed only if the workflow produced durable value:

- decision
- preference
- pattern
- correction
- reusable process
- lesson learned

Store in the correct memory type:

- project memory
- workflow memory
- reflection memory
- archived memory

## Acceptance Test

The workflow passes when Wayne can give one rough idea and Axis returns:

- a structured plan
- one practical execution path
- the correct review requirement
- a next action
- a memory/tracker update decision

## Example

Input:

```text
I want to turn the AI audit into the front-door offer for SMEs.
```

Expected loop:

```text
Capture idea
-> classify as ORCHESTRATE / BUILD
-> connect to SF&W Project Solutions
-> check current offers and pricing authority
-> create front-door offer plan
-> review claims and client risk
-> update active workstream if approved
-> capture durable decision in memory
```
