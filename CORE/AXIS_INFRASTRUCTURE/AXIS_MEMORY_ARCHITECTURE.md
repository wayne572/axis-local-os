# Axis Memory Architecture

Status: active - v4 memory taxonomy
Purpose: Define what Axis remembers, where it belongs, and when it becomes authority.

## Core Principle

Memory is operational infrastructure, not a dumping ground.

Axis remembers only information that improves future decisions, delivery, continuity, or governance.

## Memory Types

| Memory Type | Purpose | Default Location |
|---|---|---|
| short-term memory | Current session context and immediate working state | conversation / current task only |
| project memory | Durable truth about a venture, client, product, or workstream | project folders, `business/`, trackers |
| workflow memory | Reusable process, SOP, template, checklist, or operating pattern | `CORE/`, `PRODUCT_MANUAL/`, project workflow folders |
| reflection memory | Lessons learned from reviews, mistakes, outcomes, and repeated patterns | `CORE/LEARNING/`, `business/MEMORY.md`, review logs |
| archived memory | Superseded or historical material kept for reference | `ARCHIVE_NOTES/`, backups, locked shipped folders |

## Memory Authority

Authority order:

1. Current user instruction, unless unsafe or non-compliant
2. Locked governance and pricing files
3. Active business and project files
4. Current trackers
5. Reviewed memory and learning notes
6. Archive material

Archive material is not current authority unless Wayne explicitly revives it.

## Capture Rules

Capture memory when it is:

- a confirmed decision
- a durable preference
- a repeated pattern
- a useful correction
- a reusable process improvement
- a project state change
- a governance-relevant event

Do not capture:

- throwaway thoughts
- duplicated notes
- sensitive details without need
- unreviewed speculation
- temporary mood or context
- old material that conflicts with active authority

## Memory Update Timing

Memory updates occur after validation, not before.

```text
Work happens
-> output is reviewed if needed
-> durable learning is identified
-> memory update is proposed
-> Wayne approves or implicitly confirms
-> memory is stored in the correct layer
```

## Reflection Memory

Reflection memory records what the system learned from use.

It should include:

- what happened
- what worked
- what failed or created friction
- the lesson
- the rule or workflow implication
- whether an update is approved

Reflection may recommend change. It may not silently rewrite authority.

## Archived Memory

Archived memory must be clearly marked.

Use archive material for:

- comparison
- recovery
- historical context
- proof of earlier decisions

Do not use archive material for:

- pricing
- current delivery promises
- active workflows
- current client commitments

## Success Condition

Axis memory is working when Wayne feels continuity without clutter:

- the system remembers what matters
- ignores what does not
- updates slowly and deliberately
- can explain which source is current authority
