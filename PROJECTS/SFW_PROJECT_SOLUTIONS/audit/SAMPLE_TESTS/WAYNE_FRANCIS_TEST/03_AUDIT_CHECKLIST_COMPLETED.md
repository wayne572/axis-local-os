## 2. Workflow Review

### Workflow 1

Name:
Multi-channel enquiry capture

Purpose:
Capture all incoming opportunities in one place.

Current steps:
1. Enquiry arrives by email, LinkedIn, WhatsApp, or referral.
2. Wayne replies manually.
3. Follow-up may be remembered or written elsewhere.

Tools used:
Email, LinkedIn, WhatsApp, notes.

Pain points:
- Leads are scattered.
- Follow-up depends on memory.
- No consistent intake record.

Manual repetition:
- Copying lead details.
- Checking multiple inboxes.
- Writing similar first replies.

Delay points:
- When messages arrive outside the main inbox.
- When a reply needs a next action but is not logged.

Data copied between systems:
- Contact details, enquiry context, next action.

Automation opportunity:
Create a lead capture and follow-up tracker with a standard intake format.

Impact if improved:
Fewer missed replies and clearer daily follow-up.

### Workflow 2

Name:
Discovery call notes to proposal

Purpose:
Turn call notes into a clear proposal.

Current steps:
1. Discovery call happens.
2. Notes are captured inconsistently.
3. Proposal is drafted manually.
4. Follow-up is sent manually.

Tools used:
Notes, documents, email.

Pain points:
- Notes sit in different places.
- Proposal drafting takes longer than needed.
- Key facts may be missed.

Manual repetition:
- Rewriting the same proposal sections.
- Turning rough notes into scope.

Delay points:
- After call, before proposal.

Data copied between systems:
- Pain points, scope, deliverables, price, next step.

Automation opportunity:
Use a discovery notes template that feeds an implementation proposal template.

Impact if improved:
Faster proposals and cleaner scope.

### Workflow 3

Name:
Client onboarding and delivery handover

Purpose:
Move a signed client into delivery without losing detail.

Current steps:
1. Client says yes.
2. Onboarding details are collected manually.
3. Delivery actions are created manually.
4. Handover depends on notes and memory.

Tools used:
Email, documents, notes.

Pain points:
- Onboarding is too manual.
- No consistent handover checklist.
- Delivery state may be unclear.

Manual repetition:
- Asking for standard client details.
- Creating project tasks.
- Explaining next steps.

Delay points:
- Between proposal acceptance and delivery setup.

Data copied between systems:
- Client context, scope, tools, contacts, agreed deliverables.

Automation opportunity:
Use a client setup trigger and handover workflow based on the Client OS Template.

Impact if improved:
Cleaner onboarding and fewer dropped details.

## 5. Recommended First Build

Best first workflow to improve:
Lead-to-proposal command centre.

Why this one:
It tackles the highest-risk points: missed enquiries, slow follow-up, and delayed proposals.

Expected outcome:
One place to capture leads, record discovery notes, create proposal drafts, and track next actions.

Minimum viable version:
- lead tracker
- discovery notes template
- proposal template
- follow-up email templates
- status field for each prospect

Tools likely involved:
Claude folder, email, LinkedIn, WhatsApp capture later, documents.

Client input needed:
Examples of recent enquiries, discovery notes, proposals, and onboarding messages.

Risks or dependencies:
Needs consistent use. WhatsApp and email automation may need later integration.

## 6. Audit Summary

Main findings:
- Lead intake is scattered across multiple channels.
- Follow-up and proposal drafting depend too much on memory.
- Onboarding lacks a repeatable handover process.

Quick wins:
- Use one lead tracker.
- Use structured discovery notes.
- Use proposal and handover templates.

Recommended next step:
Build a lightweight Lead-to-Proposal Command Centre and test it for 30 days.
